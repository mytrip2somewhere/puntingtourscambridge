# 03-design-system.md — Phase 3: Design System

Generic. Read site specifics from `site.config.md`. Global rules live in root `CLAUDE.md`.

## PURPOSE
A distinctive, human-feeling, mobile-first visual system that converts, does not read as
AI-built, and stays robust when content managers upload images via Sveltia. Governs every
block in Phase 4.

## 3.1 PHILOSOPHY: ANTI-TEMPLATE, EDITORIAL, HUMAN
Real visual hierarchy and deliberate asymmetry, not sterile uniform grids. Custom-feeling
components with subtle micro-variation, not repeated identical cards. Distinct type pairing.
Uneven visual rhythm. Symmetry/repetition only when content calls for it.
Skill stack: Anthropic's frontend-design skill (base), TypeUI (project spec layer),
optionally UI/UX Pro Max for font/palette. Vet any third-party skill before use.

## 3.2 REFERENCE FEEL (match, don't copy)
Nomadic Matt: trusted personal voice, content-first, author-led, affiliate links inside
useful content. Plus any references Oleg supplies (point Claude Code at them to match the
design language, never clone).

## 3.3 THREE CORE BLUEPRINTS
- Asymmetric layout anchoring: break the perfect grid (e.g. large heading left, offset
  detailed table right). Cards get subtle micro-variation in layout/border radius.
- High-contrast typographic texture: bold expressive display face for H1/H2 + clean
  neutral sans for body.
- Aspect-ratio asset handshakes: strict aspect-ratio wrappers per image slot so uploads
  can't break layout. Heroes 16/9, product cards 4/3 or 1/1.

## 3.4 IMAGE OWNERSHIP SPLIT (hard rule; also in root)
- Uploaded by Oleg/content managers (never generated): tour card images, hero images,
  below-section content images, all tour photography and editorial imagery.
- Generated by Claude (no real-photo equivalent): logo, favicon, infographics.
- A favicon (SVG) is MANDATORY on every site: generate `favicon.svg` (a simple brand mark)
  in `src/public/` and link it in the base layout head (`<link rel="icon" type="image/svg+xml">`
  + apple-touch-icon).
- Generated assets MUST carry descriptive alt text / aria-label that includes a site keyword
  where natural (infographics; a purely decorative logo beside the brand wordmark uses alt="").
- Every CMS image slot is a `{ src, alt }` object: managers set the photo AND a required,
  keyword-relevant alt text per image. Templates use `{{ img:KEY }}` (src) + `{{ alt:KEY }}`
  (alt), resolved from `src/data/images.json` at build. Seed good defaults so alt is never empty.
Never invent a tour photo or editorial image. Maps are embeds, not generated images (3.5).

## 3.4b INFOGRAPHIC SVG CRAFT (hand-authored, not AI-generated)
Data infographics are hand-authored SVG, never AI-image-generated: AI image models garble
text and exact numbers (a fabrication risk on the very figures the graphic exists to show)
and output blurry raster. SVG stays crisp at any size, tiny, accessible, and editable. Rules:
- ONE distinct visual concept per page (timeline, stylized map, journey/gradient, weighted
  breakdown, decision/eligibility flow, anatomy diagram), in the site palette + display font.
  Not a table rendered as an image; not the same graphic reused across many pages.
- COLOUR/CONTRAST GOTCHA: in SVG a `<style>` class `fill` OVERRIDES an inline `fill=`
  attribute. So never set a fill inside a reusable class and then try to override it inline
  (the class wins and you get, e.g., dark text on a dark banner). Keep `fill` OUT of shared
  classes and set it inline, or give each colour its own dedicated class. Always check text
  contrast: light text on dark fills, dark text on light.
- Keep all content inside the viewBox and within the rounded background border; leave margin
  so text never runs to the edge. Validate each SVG parses as XML before shipping.
- Real data only (Phase 1 / Grok / operator), with descriptive keyword-rich alt + a caption.

## 3.5 MAPS: EMBED-ONLY (Oleg's standing rule, 2026-06-23)
Use embedded interactive maps everywhere a map appears (homepage Location section, tour
pages, etc.), NOT static images. Accept the small Core Web Vitals cost; Oleg prefers
interactive embeds site-wide and for all future sites. Implementation: a keyless Google
Maps embed iframe (`https://maps.google.com/maps?q=<place>&z=<zoom>&output=embed`) inside a
fixed aspect-ratio `.embed` wrapper (16/9), `loading="lazy"`, descriptive `title`. Real
GeoCoordinates still feed TouristAttraction schema (Phase 5). Do not ship static map images
or the old static-map-linking-to-Google-Maps pattern.

## 3.6 IMAGE SLOT HANDLING (uploaded slots)
Each slot: fixed aspect-ratio wrapper, defined dimensions, sensible object-fit,
placeholder/fallback, required alt text (entered in Sveltia), lazy-load below the fold.
Target WebP/AVIF, defined max weight. Graceful empty state so an unfilled slot never looks
broken.

## 3.7 CONVERSION COMPONENTS (honest only)
Product cards are the primary money component, routing to the affiliate link (GYG/Viator).
No widget embeds by default. Cards show honest data only: name, uploaded image, price, real
rating, real review/booking count (attributed), short hook, clear CTA. Real aggregate
urgency only (e.g. "4,000+ travelers booked this"). Availability messaging IS allowed when
grounded in a real operator statement (e.g. the operator confirms tours run daily and virtually
always have spots) and worded as such; the availability date-picker uses this to reveal the
affiliate "check the price" CTA. What stays forbidden is an INVENTED numeric counter ("3 spots
left") with no real-time source. (Refined 2026-06-23 per Oleg + operator confirmation.)
Official GYG/Viator widgets only if Oleg explicitly requests on a page.
Cards carry micro-variation so a grid doesn't read as cloned.

## 3.8 TYPE / 3.9 COLOR / 3.10 GRID
Display face for H1/H2, neutral sans for body, modular scale, generous line height,
optimized font loading. Distinctive purposeful palette (not a default Tailwind ramp), AA
contrast, accent reserved for CTAs. Mobile-first, defined spacing scale and flexible grid
supporting asymmetry, thumb-reachable CTAs, no hover-only interactions, breakpoints defined.

## 3.11 ACCESSIBILITY
Semantic HTML, logical headings, landmarks, AA contrast, visible focus, keyboard operable,
required alt text on every image (incl. maps), descriptive links, accessible mobile menu.

## 3.12 PERFORMANCE BUDGET
Image weight ceilings, modern formats, explicit width/height (CLS protection), minimal/no
heavy JS, optimized fonts. Maps are interactive embeds (3.5) lazy-loaded below the fold, so
keep other below-the-fold JS/iframes deferred to protect CWV. CWV targets set in Phase 5.

## 3.13 BLOCK LIBRARY (handoff to Phase 4)
Style: hero, product/tour card, key-metrics table, comparison table, pros/cons, FAQ
accordion, CTA banner, trust strip, author bio box, related-tours, breadcrumb, nav, footer
(incl. social icons), map embed, infographic, info-gain block, image-slot wrappers. Each gets
states, spacing, type, micro-variation rules.
